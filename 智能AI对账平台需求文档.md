## 一、产品概述与定位

### 1.1 产品愿景

打造一款 **"会思考的对账引擎"**——不仅自动化执行对账任务，更能像资深财务主管一样理解数据语义、识别异常模式、解释差异原因、建议处理方案。

### 1.2 产品定位

| 维度 | 定位 |
|------|------|
| **目标市场** | 中大型企业（年营收 5000 万以上），财务团队 5 人以上 |
| **核心价值** | 将对账效率提升 80%+，差异发现率提升 50%+，月末结账周期缩短 60%+ |
| **技术壁垒** | LLM 驱动的模糊匹配 + 多维度异常检测 + 可解释的差异分析 |
| **商业模式** | SaaS 订阅（按对接账户数/数据量计费）+ 私有化部署 |

### 1.3 与传统对账的差异

| 传统对账 | 智能 AI 对账 |
|----------|-------------|
| 基于规则精确匹配（金额必须相等） | LLM 语义理解 + 模糊匹配（容忍小数点、汇率、时差等差异） |
| 差异靠人工逐条排查 | AI 自动归类差异 + 给出根因分析 |
| 模板固定，新场景需开发 | 自然语言描述规则，AI 自动生成匹配逻辑 |
| 被动响应，问题事后发现 | 主动预警，识别趋势性异常 |
| 只有 "匹配/不匹配" 二元结果 | 置信度评分 + 多候选推荐 + 处理建议 |

### 1.4 行业痛点：为什么市场需要智能对账

以下数据来源于对全球 300+ 财务团队的调查研究（SixthFin/Odoxa 2026、AutoRek 2024、BlackLine/FSN、Deloitte、LiveFlow 2025、Gartner 2023 等）：

#### 手工依赖症：电子表格仍是主流

| 数据 | 来源 |
|------|------|
| **67%** 的英国 250 人以上企业仍用 Excel 做账务核对 | SixthFin/Odoxa (2026) |
| **84%** 的支付公司严重依赖手工任务和电子表格 | AutoRek (2024) |
| **57%** 的金融公司通过电子表格或混合方式管理对账 | AutoRek |
| **40%** 的企业最多一半的财务数据靠手工处理 | bluQube (2025) |

#### 时间黑洞：月末关账周期过长

| 数据 | 来源 |
|------|------|
| **50%** 的财务团队需要 6 个工作日以上才能关账 | CFO.com / Ledge (2025) |
| 仅 **16%** 的团队能在 3 天内完成关账 | LiveFlow / CFO Dive (2025) |
| 千人规模企业每年消耗约 **10 万人时** 在对账上，直接人力成本 **$300-500 万** | aurum.solutions |
| 财务团队 **41%** 的时间花在收集和处理数据上，而非分析数据 | Deloitte |
| 平均现金对账每月耗时 **20-50 小时** | 多来源综合 |

#### 错误率惊人：手工流程的隐性代价

| 数据 | 来源 |
|------|------|
| **59%** 的会计师每月犯多次财务错误；**18%** 每天犯错 | Gartner (2023) |
| 复杂操作中手工对账错误率可达 **45%** | NetSuite |
| 手工数据录入每笔交易 **1.6%** 的错误率（每万笔 160 个错误） | Institute of Finance & Management |
| **~94%** 的业务电子表格包含错误 | 多来源研究 |
| **1/3** 的财务领导不信任自己关账数据的可靠性 | SixthFin/Odoxa (2026) |
| **~70%** 的财务领导曾基于不准确的财务数据做出重大决策 | BlackLine/Censuswide |

#### 数据孤岛：系统碎片化

| 数据 | 来源 |
|------|------|
| **~80%** 的财务人员将月末关账延迟归咎于等待其他系统/部门的数据 | LiveFlow / CFO Dive (2025) |
| 英国企业平均管理 **8-25** 个互不连接的系统 | Accountancy Age |
| **61%** 的组织认为总账系统过多导致混乱 | FSN/BlackLine |
| **83%** 的组织被最后一个业务单元的数据提交卡住 | FSN/BlackLine |

#### 人才流失：对账工作的隐性人力成本

| 数据 | 来源 |
|------|------|
| **74%** 的会计师承受工作相关压力 | CABA (2026) |
| **77%** 的财务专业人士经历过职业倦怠 | Deloitte |
| 美国过去 2 年约 **30 万** 会计师离职（从业人数减少 17%） | 多来源综合 |
| **97%** 的财务领导承认关账对工作负荷有负面影响；**93%** 表示影响员工积极性 | SixthFin (2026) |
| 替换一名熟练会计师的成本高达其年薪的 **30%** | 美国劳工部 |

#### 合规风险与欺诈敞口

| 数据 | 来源 |
|------|------|
| 存在实质性内控缺陷的组织欺诈发生率高出 **80-90%**，通常与手工对账相关 | ACFE |
| **80%** 的支付公司预计合规成本将继续上升 | AutoRek |
| **~80%** 的公司经常错过报告截止日期 | AutoRek |
| **79%** 的公司表示对账流程无法处理当前数据量 | AutoRek |
| Macy's 在 2024 年 12 月因对账漏洞发现 **$1.51 亿** 虚假账目（真实案例） | SEC Filing |

#### 痛点一句话总结

> **手工对账每年浪费企业百万美元级成本，消耗财务团队 40%+ 时间，产出 45% 错误率，导致 30 万美国会计师离职——而全球主流对账软件在 AI 能力上几乎全部空白。**

### 2.1 厂商分层与定价

全球对账软件市场可分为三个梯队：

| 梯队 | 代表厂商 | 年费区间 | 目标客户 | 实施周期 | 核心卖点 |
|------|---------|---------|---------|---------|---------|
| **企业级** (Enterprise) | BlackLine、Oracle ARCS、SAP ICMR、Trintech Cadency | $30K-$340K+ | 年营收 1 亿以上、多实体跨国企业 | 3-12 月 | 合规控制、SOX 审计、全流程关账 |
| **中端市场** (Mid-Market) | FloQast、Trintech Adra | $30K-$120K/年 | 年营收 5000 万-5 亿企业 | 1-6 周 | 快速部署、Excel 友好、性价比 |
| **小企业** (SMB) / 国内 | Xero、QuickBooks、用友 NCC、金蝶云星空 | $240-$1000/年 | 中小企业、国内企业 | 即时-8 周 | 易用性、本地化、与核心财务系统集成 |

### 2.3 各厂商核心能力拆解（2025-2026 更新）

#### BlackLine（全球市场领导者 | Gartner 4.7/5 | 年费 $77K-$340K）

- **2025-2026 重大更新：** 收购 WiseLayer（2025.12）扩展 Agentic AI；推出 Verity AI 平台（Verity Match、Verity Prepare）；Journals Risk Analyzer 做 ML 异常检测
- **对账引擎：** 规则驱动 + ML 置信度评分，日处理 2 亿+ 交易；**未使用 LLM 做语义匹配**
- **Agentic AI：** WiseLayer 专项 Agent（Angela 做应计/收入、Payton 做薪酬）；Verity Prepare 自动编写对账说明
- **预测分析：** 集团内部预测指导 + AR 付款预测
- **不足：** 无 LLM 语义匹配；无自然语言查询；无中国市场适配；价格昂贵；实施 3-6 月
- **借鉴点：** 工作流引擎设计、审计追踪完整度、多实体管理架构、SOX 合规控制

#### Trintech Cadency（企业级对账专业厂商 | Gartner 4.4/5 | 年费 $60K+）

- **2025-2026 重大更新：** 平台级 Agentic AI 嵌入所有模块（Match、Balance、Journal）；95% 异常处理时间缩减
- **对账引擎：** 智能规则 + AI 多向匹配（1:1/1:N/N:N），匹配率 99%+；日处理数百万笔交易
- **AI 异常检测：** ML 预测异常，主动路由到引导工作流
- **不足：** 无 LLM 语义匹配；无自然语言查询；部署复杂（4-8 月实施）
- **借鉴点：** 风险评分矩阵、自动认证流程、高吞吐交易匹配

#### AutoRek ARIA（金融服务业专业厂商 | 2025 年推出 AI Agent）

- **2025-2026 重大更新：** 推出 ARIA（多技能 AI Agent），动态学习对账模式；自动规则生成（快 90-95%）
- **对账引擎：** 匹配率 99.99%；智能异常管理 + 升级协议；专注资本市场（T+1 结算）、支付（FCA 监管）、保险（56 天结算期）
- **技术架构：** 基于 Microsoft Azure；30 年金融对账数据积累
- **不足：** 仅覆盖金融服务细分市场；无中国市场
- **借鉴点：** 规则自动生成、异常升级协议、监管合规深度

#### HighRadius（AI-Native 记录到报告 | 2026 Gartner MQ Challenger 首次入选）

- **2025-2026：** 自主 AI Agent 用于银行对账、收入对账、集团内部管理；90%+ 自动匹配，99% 准确率
- **差异化：** 自学习 Agent 从历史数据自动发现匹配规则；实时异常监控；30% 关账天数缩减
- **不足：** 无 LLM 语义匹配；无中国市场适配
- **借鉴点：** 自学习规则发现、触摸式执行、实时监控仪表盘

#### Digits（AI 会计 Agent | 2025.11 推出 AI 银行对账）

- **2025-2026：** 24/7 运行的 AI Agent：银行对账单采集+对账、交易分类、薪酬记账、工作底稿管理、方差分析
- **技术架构差异化：** **预测模型做数学（不能幻觉），LLM 仅做 Agent 编排和结果沟通**。18 个定制模型。基准测试比 GPT-4o 分类准确率高 54%。LLM 在财务分类上准确率天花板约 70%，Digits 达 mid-90%
- **自然语言：** "Ask Digits" — 自然语言发起对账、查询财务数据
- **处理速度：** 40ms/笔交易
- **不足：** SMB 定位；无企业级多实体/多币种；无中国市场
- **借鉴点：** 预测模型 vs LLM 的架构分离、40ms 级处理速度、自然语言查询

#### JustPerform / insightsoftware（Isaac AI Agent | 2025 年推出）

- **2025-2026：** Isaac AI Agent 自主完成 95% 对账任务；置信度评分（1-3 级）；无需规则配置即交付完整对账结果
- **不足：** 新产品（2025 推出），验证案例有限；无中国市场
- **借鉴点：** 零配置对账、置信度评分体系

#### Simetrik（支付/电商对账专业 | 拉美市场领导者）

##### 1.2 智能字段映射（AI 驱动）

这是本产品的核心差异化能力之一。

**传统方式痛点：** 不同来源的字段命名千差万别 —— "交易金额" vs "Transaction Amount" vs "TXN_AMT" vs "金额(元)" —— 手工映射耗时且易错。

**AI 方案：**

```
输入：源字段列表 + 目标字段列表

┌─────────────────────────────────────────────────┐
│ AI 字段语义映射引擎                              │
│                                                 │
│ 源字段: "TXN_AMT"                               │
│   ↓ 语义理解（LLM）                              │
│   → "这是一个交易金额字段"                        │
│   → 字段类型推断: Decimal(18,2)                  │
│   → 建议映射: "交易金额" (置信度: 98%)            │
│   → 数据格式识别: "1000.00" → 有小数点, 2 位小数  │
│                                                 │
│ 源字段: "VALUE_DATE"                            │
│   ↓ 语义理解（LLM）                              │
│   → "这是一个起息日/入账日期字段"                  │
│   → 建议映射: "入账日期" (置信度: 95%)            │
│   → 日期格式识别: "20240115" → yyyyMMdd          │
└─────────────────────────────────────────────────┘
```

**功能要点：**
- LLM 语义分析自动识别字段含义
- 支持 20+ 语言字段名映射
- 字段格式自动识别与转换规则推荐
- 人工确认/修正界面（一键接受所有推荐）
- 映射方案保存为模板，同类数据源复用

##### 1.3 数据预览与清洗

- 数据源连接后即时预览（前 100 行）
- 自动数据质量检查：
  - 必填字段缺失率
  - 金额字段格式异常
  - 日期范围异常
  - 重复行检测
  - 金额汇总与源系统校验
- 一键数据清洗：
  - 去除空白行/特殊字符
  - 金额格式化（"1,000.00" → 1000.00）
  - 日期标准化
  - 负号/借貸方向统一

##### 1.4 定时同步策略

- Cron 表达式调度
- 事件驱动同步（接收到新文件 / Webhook 触发）
- 同步失败自动重试 + 告警
- 同步历史与耗时监控

#### 模块 2：对账规则引擎

##### 2.1 预置规则模板

| 规则模板 | 适用场景 | 匹配逻辑 |
|----------|---------|----------|
| 银行对账-标准 | 银行流水 vs ERP 日记账 | 金额+日期(±1天)+摘要模糊匹配 |
| 银行对账-严格 | 高风险账户 | 金额+日期+流水号精确匹配 |
| 三方支付对账 | 支付流水 vs 结算单 | 金额+手续费拆解+订单号匹配 |
| 应收对账 | 我方应收 vs 客户确认 | 发票号+金额+客户名称 |
| 应付对账 | 供应商账单 vs 我方采购 | PO号+发票号+金额 |
| 跨系统核对 | 总账 vs 子账 | 科目+期间+金额 |
| 集团内部往来 | 关联交易双方 | 交易编号+金额+方向 |

##### 2.2 可视化规则编排

- 拖拽式规则构建器（类似决策表编辑器）
- 匹配条件组合：AND/OR/NOT 逻辑
- 支持的条件类型：
  - 精确匹配 / 范围匹配 / 包含匹配 / 正则匹配
  - 金额容差（绝对值/百分比）
  - 日期偏移（±N 天/工作日）
  - 金额汇总匹配（一对多）
- 规则优先级排序
- 规则测试沙箱：使用历史数据验证规则效果

##### 2.3 自然语言规则生成（AI 驱动）

**传统方式痛点：** 财务人员无法直接配置规则，必须依赖 IT 人员翻译。

**AI 方案：**

```
用户输入（自然语言）：
"把金额相差不超过5块钱，日期在前后2个工作日内，
且摘要里有相同供应商名称的记录自动匹配上。
如果有手续费差异，单独标记为'手续费差异'。"

  ↓ LLM 解析 + 规则生成

生成的规则（结构化）：
{
  "name": "供应商付款对账-含手续费容差",
  "conditions": [
    { "field": "amount_diff_abs", "operator": "lte", "value": 5 },
    { "field": "date_diff_business_days", "operator": "lte", "value": 2 },
    { "field": "supplier_name_similarity", "operator": "gte", "value": 0.8 }
  ],
  "exceptions": [
    { "condition": "fee_identified", "tag": "手续费差异" }
  ]
}
```

**功能要点：**
- 自然语言输入 → 结构化规则输出
- LLM 转译过程可视化（用户可审查每一步）
- 规则冲突检测（新规则与已有规则是否有逻辑冲突）
- 规则有效性模拟（AI 预测新规则的匹配率和误匹配率）

##### 2.4 规则推荐与优化（AI 驱动）

- 基于历史对账结果，AI 推荐：
  - "这类未匹配记录如果将日期容差从 1 天放宽到 2 天，可自动匹配 87%"
  - "摘要中包含品牌名的记录，建议增加供应商名称匹配规则"
- 规则效果持续监控 → 建议优化/废弃低效规则

#### 模块 2.5：自定义对账（新增）

面向「非标准场景」与「多源组合对账」需求，用户可自行组合数据来源与匹配规则，保存为可复用方案并一键执行，无需依赖固定任务模板。

##### 2.5.1 自定义对账方案定义

| 字段 | 说明 |
|------|------|
| 方案名称 / 编码 | 组织内唯一编码，便于复用与审计 |
| 期间类型 | 日 / 月 / 自定义期间 |
| 默认期间 | 如 `2026-07`，执行时可覆盖 |
| 状态 | 草稿 / 启用 / 停用 |
| 描述 | 业务说明 |

##### 2.5.2 多数据源组合配置

- **主数据源 A**：一条（通常为 ERP / 本方账）
- **对账方 B**：可多选，支持同时对多个来源
- **数据源类型不限于**：银行 API、HTTP 接口、三方支付、ERP、文件导入、数据库直连、手工录入等
- 多 B 源场景：系统为每个 B 源创建独立 `CUSTOM` 类型对账任务（A vs Bᵢ），统一归属本方案

##### 2.5.3 规则绑定与匹配层选择

- 可勾选本方案专用规则 ID 列表；不选则使用组织下全部启用规则
- 匹配引擎按任务 `ruleIds` **精确过滤**，避免「自定义」仍套用全部组织规则
- 可配置启用层：精确匹配 / 规则匹配 / AI 语义 / 拆单匹配

##### 2.5.4 方案预检与一键执行

**预检内容：**
- 数据源是否存在且启用
- A/B 是否冲突（同一源不能同时作 A 与 B）
- 待对账 `PENDING` 记录数量提示
- 所选规则是否存在/启用

**执行：**
- 支持同步 / 异步
- 可覆盖期间、临时规则、临时 B 源列表
- 记录 `last_run_task_id` / `last_run_at`，草稿方案首次执行后自动启用

##### 2.5.5 向导式配置与结果跳转

四步向导：**基本信息 → 选择数据源 → 配置规则 → 确认保存**
执行后跳转匹配审核页，差异进入差异管理中心，复用现有审核与处理链路。

##### 2.5.6 自然语言创建方案（AI 驱动）

用户无需手动填写四步向导，可直接用自然语言描述对账需求，AI 自动解析并生成方案配置：

| 能力 | 说明 |
|------|------|
| 自然语言输入 | 例如：「对账工商银行流水和SAP系统数据，按金额精确匹配，日期允许1天差异」 |
| AI 解析 | 调用 LLM 将自然语言转化为结构化方案（名称、A/B 源、规则、期间、匹配层） |
| 名称→ID 解析 | 后端将 AI 返回的数据源/规则名称，通过模糊匹配映射到数据库实际 ID |
| 预填向导 | 解析结果通过四步向导展示，用户可确认或修改后保存 |
| 未匹配提示 | 无法匹配的数据源/规则名称显式警告，引导用户手动选择 |

**处理流程：**
```
用户输入自然语言 → POST /nl-parse → AI 解析为方案 JSON
→ 后端名称→ID 模糊匹配 → 返回预填方案
→ sessionStorage 传递 → 跳转四步向导（表单已预填）
→ 用户确认/修改 → 保存方案
```

**API：** `POST /api/v1/custom-recon/definition/nl-parse`
- 请求：`{ "description": "...", "orgId": 1 }`
- 响应：`{ definition: { ...预填方案 }, unresolvedSources: [], unresolvedRules: [], aiExplanation: "..." }`

**模糊匹配策略（3 步 fallback）：**
1. 精确匹配（忽略大小写和空格）
2. 子串包含匹配
3. 中文关键词匹配（至少 2 个关键词命中）

**API 前缀：** `/api/v1/custom-recon/definition`（分页 / CRUD / 启用停用 / validate / run）

#### 模块 3：智能匹配引擎

##### 3.1 分层匹配架构

```
                    第 1 层：精确匹配
        金额 = 金额 AND 流水号 = 流水号 AND 日期 = 日期
                    匹配率目标: 60-70%
                         ↓ 未匹配
                    第 2 层：规则匹配
        金额容差 ≤ 阈值 AND 日期偏移 ≤ N天 AND 摘要相似度 ≥ 80%
                    匹配率目标: 15-25%
                         ↓ 未匹配
                    第 3 层：AI 语义匹配
        LLM 理解交易语义 + 向量相似度 + 上下文推理
                    匹配率目标: 5-10%
                         ↓ 未匹配
                    第 4 层：差异管理
                自动分类 + 根因分析 + 处理建议
```

##### 3.2 AI 语义匹配详解

这是本产品的核心技术壁垒。

**传统匹配失败的原因：**
- 同一笔交易的描述文本完全不同：ERP 记"采购付款-甲公司 2024Q1"，银行显示"跨行转账 甲科技有限公司 CN20240115-005"
- 一笔对多笔的拆单场景：ERP 一次付款 100 万，银行分 3 笔（50万+30万+20万）扣款
- 手续费拆分：交易金额 = 本金 + 手续费，两边记录金额不同
- 汇率差异：外币交易的记账汇率与实际结算汇率不同
- 时间差异：ERP 按交易日期记账，银行按清算日期（T+1/T+2）

**AI 语义匹配流程：**

```
┌─────────────────────────────────────────────────────┐
│              AI 语义匹配引擎                          │
│                                                     │
│  输入: 未匹配记录 A (ERP) + 候选记录 B1..Bn (银行)    │
│                                                     │
│  Step 1: 语义向量化                                  │
│    将交易摘要、对方户名、用途等文本用 Embedding        │
│    模型向量化 → cosine_similarity 计算语义距离        │
│                                                     │
│  Step 2: 多维度打分 (LLM)                            │
│    ┌───────────────────────────────────────┐        │
│    │ 维度           │ 权重  │ A↔B1 │ A↔B2  │        │
│    ├────────────────┼───────┼──────┼───────┤        │
│    │ 金额匹配度       │ 30%  │ 0.95 │ 0.70  │        │
│    │ 日期匹配度       │ 20%  │ 0.90 │ 0.90  │        │
│    │ 交易方语义相似度  │ 25%  │ 0.88 │ 0.40  │        │
│    │ 用途语义相似度    │ 15%  │ 0.92 │ 0.30  │        │
│    │ 流水号模糊匹配    │ 10%  │ 0.50 │ 0.60  │        │
│    │ 综合加权得分      │      │ 0.89 │ 0.60  │        │
│    └───────────────────────────────────────┘        │
│                                                     │
│  Step 3: 最佳匹配决策                                │
│    综合得分 ≥ 0.85 → 自动匹配                        │
│    综合得分 0.70-0.85 → 推荐匹配（人工确认）          │
│    综合得分 < 0.70 → 标记为差异                      │
│                                                     │
│  Step 4: 匹配解释（可解释 AI）                        │
│    "A 与 B1 匹配得分 0.89：金额相等(0.95)、            │
│     日期相差1天(0.90)、对方户名语义高度相关(0.88)..."  │
└─────────────────────────────────────────────────────┘
```

##### 3.3 一对多/多对一拆单匹配

**场景：**
- ERP 付款 10,000 元 → 银行分 6,000 + 4,000 两次扣款（跨日）
- 电商日汇总收款 50,000 元 → 对应的 300 笔订单明细

**算法：**
- 子集和问题（Subset Sum）优化：在候选记录中搜索金额组合
- AI 辅助：LLM 分析拆单原因（是否有手续费、是否跨境分笔），缩小搜索空间
- 性能：100 条候选记录内，30 秒内完成组合搜索

##### 3.4 匹配置信度评分

- 每对匹配记录附带置信度评分（0-100%）
- 匹配质量仪表盘：按置信度区间分布展示
- 低置信度匹配自动提交人工审核
- 连续低置信度的数据源自动触发规则优化建议

#### 模块 4：差异管理中心

##### 4.1 差异自动分类（AI 驱动）

AI 将差异自动归入以下类别，无需人工逐条判断：

| 差异类别 | 典型表现 | AI 识别依据 |
|----------|---------|------------|
| **时间差** | 金额相同但日期跨月 | 金额精确匹配 + 日期差异 + 跨月 |
| **手续费差异** | 金额差几元/几十元 | 差值在常见手续费范围内（0.1%-1%） |
| **汇率差异** | 外币交易金额不同 | 币种字段 + 差异比例接近汇率波动范围 |
| **人为录入错误** | 金额数字颠倒/错位 | 数字相似度分析（如 "1234" vs "1243"） |
| **未达账项** | 一方有记录一方无 | 单边存在 + 时近期末 |
| **重复记账** | 同金额同日期出现两次 | 完全重复检测 + 大概率是重复 |
| **对方未入账** | 仅我方有记录 | 单边 + 非期末时间点 |
| **未知差异** | 无法自动归类 | 提交人工判断并学习 |

##### 4.2 根因分析（AI 驱动）

**AI 不仅告诉你"有差异"，还告诉"为什么"：**

```
差异编号: #D20240715-0032
金额差异: ¥1,200.00

AI 根因分析结果:
┌─────────────────────────────────────────────┐
│ 🔍 差异根因: 汇率结算差                       │
│                                             │
│ 📋 分析过程:                                  │
│ 1. 该交易为USD结算 (ERP记录汇率 7.1000)       │
│ 2. 银行实际结算汇率 7.0824                    │
│ 3. 折算差异: $10,000 × (7.1000-7.0824)      │
│    = ¥176.00 (汇率差)                        │
│ 4. 另含 ¥24.00 跨境手续费                     │
│ 5. 剩余 ¥1,000.00 未匹配 → 可能是对方手续费    │
│                                             │
│ 💡 建议:                                      │
│ • 将 ¥176.00 计入"汇兑损益"科目               │
│ • 将 ¥24.00 计入"手续费"科目                  │
│ • ¥1,000.00 联系对方银行确认                  │
│                                             │
│ ⚠️ 风险等级: 🟡 中等 (涉及外币, 需财务主管审核)  │
└─────────────────────────────────────────────┘
```

##### 4.3 差异处理建议（AI 驱动）

- 基于差异类别 + 历史处理记录，AI 给出处理方案
- 常见差异一键生成调整分录
- 调整分录可预览 → 确认 → 自动写入 ERP
- AI 学习用户处理模式，建议越来越精准

##### 4.4 差异处理工作流

- 差异分配：指定处理人 / 自动按规则分配
- 处理状态：待处理 → 处理中 → 待审核 → 已完成 → 已关闭
- 处理时限：SLA 配置 + 超时自动升级
- 批量处理：同类差异可一键批量操作
- 处理记录完整留痕（操作人、时间、处理方式、调整分录）

## 五、AI 能力矩阵

### 5.0 架构原则：预测模型 vs LLM 的分层设计

在深入 AI 能力矩阵之前，一个关键的架构决策需要明确。基于对 Digits（预测模型 + LLM）、Docyt（融合架构）和传统厂商（纯规则/ML）的深入分析，**业界最佳实践已收敛到以下范式：**

```
┌─────────────────────────────────────────────────────┐
│              AI 能力分层架构                          │
│                                                     │
│  第 3 层：LLM 层（沟通 & 推理）                       │
│  ┌─────────────────────────────────────────────┐    │
│  │ • 自然语言查询/规则生成                        │    │
│  │ • 语义匹配（多维度打分 + 解释）                 │    │
│  │ • 根因分析 (Chain-of-Thought)                │    │
│  │ • 报告生成 / 处理建议解释                      │    │
│  │ ⚠️ LLM 在财务分类上准确率天花板约 70%           │    │
│  │    → 仅用于理解、解释、推理，不用于计算          │    │
│  └─────────────────────────────────────────────┘    │
│                        ↑                            │
│  第 2 层：预测模型层（计算 & 分类）                    │
│  ┌─────────────────────────────────────────────┐    │
│  │ • 金额匹配度计算 (数值比较)                     │    │
│  │ • 差异分类 (微调模型，准确率 mid-90%)           │    │
│  │ • 异常检测 (Isolation Forest / Prophet)       │    │
│  │ • 置信度评分 (规则 + 统计)                     │    │
│  │ ✅ 数学计算 → 不能幻觉 → 预测模型/统计算法       │    │
│  └─────────────────────────────────────────────┘    │
│                        ↑                            │
│  第 1 层：规则引擎层（确定性匹配）                      │
│  ┌─────────────────────────────────────────────┐    │
│  │ • 精确匹配 (金额 + 日期 + 流水号)               │    │
│  │ • 规则匹配 (金额容差 + 日期偏移 + 正则)          │    │
│  │ • 拆单组合搜索 (Subset Sum)                   │    │
│  │ ✅ 确定性逻辑 → 零幻觉 → 传统代码实现            │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

> **核心原则：** 数学计算交给预测模型/规则引擎（不能出错），语义理解和推理交给 LLM（不能计算）。这一架构在 Digits 的实战中获得验证——预测模型比 GPT-4o 在财务分类上准确率高 54%。

### 5.1 AI 能力全景（含竞争对标）

| AI 能力 | 技术方案 | 应用模块 | 成熟度 | 差异化程度 | 是否有竞品实现 |
|---------|---------|---------|--------|-----------|--------------|
| **智能字段映射** | LLM 语义理解 + Embedding | 数据源管理 | ⭐⭐⭐ | 🟢 高 | Simetrik 有 AI 建议映射 |
| **自然语言规则生成** | LLM + 约束解码 | 规则引擎 | ⭐⭐⭐ | 🟢 极高 | ❌ 无竞品 |
| **LLM 语义匹配** | Embedding + LLM 多维度打分 | 匹配引擎 | ⭐⭐⭐⭐ | 🟢 极高 | ❌ 无竞品（所有厂商用 ML/规则） |
| **拆单组合匹配** | 子集和算法 + LLM 辅助 | 匹配引擎 | ⭐⭐⭐ | 🟡 中高 | 多家有 1:N 匹配（非 LLM） |
| **差异自动分类** | 微调小模型 + Few-shot LLM | 差异管理 | ⭐⭐⭐⭐ | 🟢 高 | Digits/Trintech 有 ML 分类 |
| **根因分析 (CoT)** | LLM 推理链 | 差异管理 | ⭐⭐⭐ | 🟢 极高 | ❌ 无竞品 |

| **持续学习优化** | 人工反馈 → 模型微调 | 全局 | ⭐⭐ | 🟡 中高 | Trintech/HighRadius 有自学习 |

### 5.2 AI 模型选型建议

| 场景 | 推荐模型（国内优先） | 备选 | 原因 |
|------|---------------------|------|------|
| **字段语义映射** | Qwen-Turbo / DeepSeek-Chat | GPT-4o mini | 高精度文本理解，成本可控 |
| **语义匹配打分** | 本地 Embedding (BGE-M3) + DeepSeek-Reasoner / Qwen-Max | Claude Sonnet | Embedding 粗排，LLM 精排 |
| **自然语言规则解析** | Qwen-Plus / DeepSeek-Chat | Claude Sonnet / GPT-4o | 需要严格的结构化输出 |
| **根因分析 (CoT)** | DeepSeek-Reasoner / Qwen-Max / Kimi-32k | Claude Opus / GPT-4o | 需要深度推理链 |
| **自然语言查询** | Qwen-Plus / DeepSeek-Chat | Claude Sonnet + Text-to-SQL | 平衡准确性与延迟 |
| **差异分类** | Qwen-Turbo / DeepSeek-Chat | 微调小模型 | 高频调用，需低成本低延迟 |
| **异常检测** | 统计模型 (Prophet/Isolation Forest) | — | 数值分析优于 LLM |
| **报告生成** | Qwen-Plus / Moonshot-v1-8k | GPT-4o mini | 结构化中文文本生成 |

> **部署说明：** 平台默认 `ai.llm.provider=mock`；生产环境推荐优先选用 DeepSeek / 通义千问 / Kimi 等国内大模型，满足数据合规与中文场景；亦可通过 `custom` 接入企业私有化 OpenAI 兼容网关。

### 5.3 AI 可靠性与安全

- **可解释性：** 所有 AI 决策附带可读的解释文本
- **人机协同：** AI 做推荐，人类做决策（Human-in-the-loop）
- **置信度阈值：** 低于阈值自动转人工
- **幻觉防控：** 金额相关输出二次校验；结构化约束输出
- **数据隐私：** 支持本地化部署 LLM，敏感数据不出企业网络
- **审计可追溯：** 所有 AI 推荐/决策记录完整日志

### 7.2 核心表结构概要

```sql
-- 对账任务
CREATE TABLE recon_task (
    id UUID PRIMARY KEY,
    org_id UUID NOT NULL,
    name VARCHAR(200),
    source_a_id UUID NOT NULL,       -- 数据源 A
    source_b_id UUID NOT NULL,       -- 数据源 B
    rule_config JSONB,               -- 规则配置
    period VARCHAR(20),              -- 对账期间 (2024-07)
    status VARCHAR(20),              -- pending/running/completed/error
    match_summary JSONB,             -- 匹配汇总统计
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 对账匹配结果
CREATE TABLE recon_match (
    id UUID PRIMARY KEY,
    task_id UUID REFERENCES recon_task(id),
    record_a_id UUID,                -- 数据源 A 的记录
    record_b_id UUID,                -- 数据源 B 的记录
    match_type VARCHAR(30),          -- exact/rule/ai_semantic/ai_split/manual
    confidence DECIMAL(5,2),         -- 置信度 0.00-100.00
    match_dimensions JSONB,          -- 各维度打分详情
    ai_explanation TEXT,             -- AI 匹配解释
    status VARCHAR(20),              -- auto_confirmed/pending_review/manual_confirmed
    created_at TIMESTAMP DEFAULT NOW()
);

-- 差异记录
CREATE TABLE recon_discrepancy (
    id UUID PRIMARY KEY,
    task_id UUID REFERENCES recon_task(id),
    record_id UUID,                  -- 未匹配的记录
    side VARCHAR(10),                -- source_a / source_b
    category VARCHAR(50),            -- 差异类别 (AI分类)
    ai_root_cause TEXT,              -- AI 根因分析
    ai_suggestion JSONB,             -- AI 处理建议
    amount_diff DECIMAL(18,2),
    risk_level VARCHAR(10),          -- low/medium/high/critical
    handler_id UUID,                 -- 指定处理人
    status VARCHAR(20),              -- pending/processing/resolved/closed
    resolved_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 数据源原始记录
CREATE TABLE raw_record (
    id UUID PRIMARY KEY,
    source_id UUID REFERENCES data_source(id),
    batch_id VARCHAR(100),
    raw_data JSONB,                  -- 原始数据
    normalized_data JSONB,           -- 标准化后数据
    amount DECIMAL(18,2),
    currency VARCHAR(3),
    transaction_date DATE,
    transaction_ref VARCHAR(500),    -- 交易参考号/流水号
    description TEXT,                -- 摘要/描述
    counter_party VARCHAR(300),      -- 对方名称
    direction VARCHAR(10),           -- debit/credit
    hash_value VARCHAR(64),          -- 数据指纹（去重用）
    created_at TIMESTAMP DEFAULT NOW()
);

-- AI 推理日志（审计用）
CREATE TABLE ai_inference_log (
    id UUID PRIMARY KEY,
    module VARCHAR(50),              -- field_mapping/semantic_match/root_cause/...
    input_data JSONB,
    output_data JSONB,
    model_used VARCHAR(100),
    tokens_used INT,
    latency_ms INT,
    confidence DECIMAL(5,2),

## 十、附录：核心算法说明

### 10.1 AI 语义匹配算法

```
算法：多阶段语义匹配

输入：
  - 未匹配记录集合 U = {u₁, u₂, ..., uₙ}（来自数据源 A）
  - 候选记录集合 C = {c₁, c₂, ..., cₘ}（来自数据源 B）
  - 日期窗口 W（默认 ±7 天）
  - 金额容差 T（默认 ±1%）

输出：
  - 匹配对列表 M = {(uᵢ, cⱼ, score, explanation)}

流程：
  Stage 1: 候选筛选（倒排索引）
    对每条 uᵢ:
      1. 按日期窗口 W 筛选 → 候选集 C'
      2. 按金额容差 T 筛选 → 候选集 C''
      3. 按交易方哈希索引粗排 → Top K 候选

  Stage 2: 向量相似度粗排（Embedding）
    对每条 uᵢ 的 Top K 候选:
      1. 拼接文本: "[交易方] [摘要] [金额] [日期]"
      2. BGE-M3 → 向量 v(uᵢ), v(cⱼ)
      3. cosine_similarity(v(uᵢ), v(cⱼ)) → 排序
      4. 保留 Top 3 进入精排

  Stage 3: LLM 多维度精排
    对每条 uᵢ 的 Top 3 候选:
      Prompt:
        "你是一个财务对账专家。请评估以下两条记录是否为同一笔交易，
        从以下维度打分（0-1）：
        - 金额匹配度
        - 日期匹配度
        - 交易方名称语义相似度
        - 交易摘要语义相似度
        - 交易参考号匹配度
        
        记录 A: {uᵢ 的 JSON}
        记录 B: {cⱼ 的 JSON}
        
        返回 JSON 格式的评分和判断理由。"

      解析 LLM 返回的评分 → 加权综合得分
      Weight = [0.30, 0.20, 0.25, 0.15, 0.10]

  Stage 4: 决策
    综合得分 ≥ 0.85 → 自动匹配
    综合得分 0.70-0.85 → 推荐匹配
    综合得分 < 0.70 → 标记差异

  Stage 5: 拆单补充（可选）
    对仍未匹配的 uᵢ，尝试在 C 中搜索子集和组合：
      subset_sum(target=uᵢ.amount, candidates=C.amounts, tolerance=T)
      若有解 → 标记为"一笔对多笔"匹配
```

### 10.2 差异根因分析算法

```
算法：基于 LLM 思维链的差异根因分析

输入：
  - 差异记录 d（含源记录和目标候选）
  - 上下文：账户类型、交易历史、行业类型

流程：
  1. 特征提取：
     - 金额差异的数值特征（绝对值、百分比、是否整数）
     - 日期差异特征（日历日/工作日、是否跨月/季/年）
     - 文本相似度特征（编辑距离、语义相似度）
     - 结构性特征（币种、账户类型、交易类型）

  2. LLM 推理链 (Chain-of-Thought):
     Prompt:
     "你是一个资深财务审计师。请分析以下对账差异的根因：

     差异信息：
     - ERP记录: {json}
     - 银行记录: {json}
     - 账户类型: {account_type}
     
     请按以下步骤推理：
     Step 1: 金额差异的数值特征是什么？
     Step 2: 这些特征可能对应什么类型的差异？
     Step 3: 结合交易上下文，最可能的根因是什么？
     Step 4: 需要哪些额外信息来确认？
     Step 5: 给出处理建议。
     
     以 JSON 格式输出完整分析。"

  3. 结果校验：
     - 差异类别是否在已知类别中
     - 建议是否可操作
     - 置信度是否合理

  4. 反馈闭环：
     - 用户确认/修正 → 存入标注数据
     - 定期微调分类模型
```

### 10.3 自然语言查询执行流程

```
用户: "上个月与工商银行的差异中，金额超过1万且未解决的有多少？"

Step 1: LLM 意图理解 + 实体识别
  → intent: "统计查询"
  → entities: { bank: "工商银行", period: "上个月", amount: ">10000", status: "未解决" }

Step 2: Text-to-SQL 生成
  → SELECT COUNT(*) FROM recon_discrepancy d
    JOIN recon_task t ON d.task_id = t.id
    JOIN account a ON t.account_id = a.id
    WHERE a.bank_name = '工商银行'
      AND t.period = '2024-06'
      AND ABS(d.amount_diff) > 10000
      AND d.status IN ('pending', 'processing')

Step 3: SQL 校验与执行
  → 语法校验 → 权限校验 → 执行 → 结果: 8 笔

Step 4: LLM 结果润色
  → "2024年6月与工商银行的对账中，金额超过1万元且尚未解决的差异共 8 笔，
     合计金额 ¥156,200.00。其中金额最大的一笔为 ¥52,000.00（供应商付款延迟）。
     需要我列出这 8 笔的详细信息吗？"
